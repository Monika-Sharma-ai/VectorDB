Product Quantization (PQ)

Split each vector into chunks, replace each chunk with a centroid ID. ~384× memory savings.

Intuition Don't store the full vector — store a code of which centroid each sub-vector is closest to.

Split768-d → 8 sub-vectors of 96-d
Clusterk-means on each sub-space (256 centroids)
Encodestore nearest centroid ID (1 byte each)
Distancelookup table + sum

Compression: 768 floats (3072 B) → 8 bytes. ~384× smaller.

IVF — Inverted File Index

Cluster all vectors into K cells. At query time, search only the closest nprobe cells.

Intuition Skip 99% of the dataset by only scanning the few cells nearest to Q.

Buildk-means on dataset → K centroids
Assigneach vector → its nearest centroid
Probefind nprobe closest cells to Q
Scanonly those cells' inverted lists

Trade-off: higher nprobe → higher recall but slower. Typical: K=4096, nprobe=8–32.

IVF + PQ — The Workhorse

IVF finds the right region. PQ compresses what's inside. Together they handle billions.

Why combine IVF alone still stores full vectors → too much RAM. PQ alone scans everything → too slow. Combined → fast and tiny.

  • Stage 1: coarse search → top nprobe cells
  • Stage 2: score PQ codes inside those cells via lookup tables
  • Stage 3 (optional): re-rank top 100 with full float vectors

Why FAISS uses it: 1B × 768 floats = 3 TB → 1B × 8 PQ bytes = 8 GB. Fits on one machine.

ScaNN — Anisotropic Quantization

Google's twist: minimize quantization error in the direction that actually matters for inner-product.

Insight Standard PQ minimizes L2 reconstruction error — but search uses inner products. Errors orthogonal to the query direction are harmless.

  • Standard PQ: uniform error ball around each centroid
  • ScaNN: elongated error — squeezed along the query direction, looser elsewhere
  • Result: ~2× recall at the same latency for cosine / IP search

Use when: very large corpora with cosine / inner-product (most embedding models).
