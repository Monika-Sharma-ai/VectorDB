1. What is Milvus?

Milvus is an open-source, distributed vector database built by Zilliz. It is designed to scale from a laptop (Milvus Lite) all the way to a Kubernetes cluster searching tens of billions of vectors with GPU acceleration.

🪐 Analogy — A Galactic Postal Service:
Imagine a postal service that can deliver messages anywhere in the galaxy.
  • One small office (Milvus Lite) handles your village.
  • A national network (Milvus Standalone) covers a whole country.
  • A galactic federation (Milvus Distributed) coordinates planets, with separate departments for receiving, sorting, querying, and storing — all working in parallel.
Same address book, same envelopes — only the org chart grows. That's Milvus.

2. Milvus's Core Concepts

  • 📚 Collection — like a SQL table, but for vectors + scalar fields.
  • 🧱 Schema — defines field names, types, and which is the vector field.
  • 🧩 Partition — physical split inside a collection (e.g., per date or tenant) for faster filtering.
  • 🎯 Entity — one row: id + vector + scalar fields.
  • 📐 Index — the data structure that makes search fast (HNSW, IVF_FLAT, DiskANN…).
  • 📏 Metric — L2, IP (inner product), or COSINE.

3. Milvus Architecture (the Big Picture)

Milvus separates compute from storage — a "shared-nothing" design where each role can scale independently.

🏭 Analogy — A Modern Factory:
The access layer = reception desk taking orders. The coordinator = factory manager dispatching work. Data / Index / Query nodes = specialized assembly lines (one packs raw goods, one builds the product, one fulfills orders). Object storage = the warehouse holding everything long-term. Need more orders served? Just add more query nodes — the rest doesn't care.

5. How Searching Actually Happens

1
Client sends a search to the proxy: vector + top_k + optional scalar filter expression like year >= 2010 AND genre == "sci-fi".
2
Proxy routes to the right collection and forwards to Query Nodes holding that data.
3
Each query node runs ANN search on its index segment in parallel, applying scalar filters first when possible.
4
Partial results stream back to the proxy, which merges and re-ranks them globally.
5
Top-K matches are returned with IDs, distances, and any requested output fields.
📦 Analogy — Amazon's Search Bar:
You search "blue running shoes". Amazon doesn't ask one warehouse — every regional warehouse searches its inventory at the same time, returns top picks, and a central system merges them. Milvus does this for vectors.

7. Real Python Example

from pymilvus import MilvusClient, DataType

# 1. Connect — Milvus Lite (file) or full server
client = MilvusClient("milvus_demo.db")          # or uri="http://localhost:19530"

# 2. Create a collection with a schema
client.create_collection(
    collection_name="movies",
    dimension=384,
    metric_type="COSINE",
)

# 3. Insert entities (vectors + scalar fields)
client.insert(
    collection_name="movies",
    data=[
        {"id": 1, "vector": [0.12, -0.44, ...],
         "title": "Inception", "year": 2010, "genre": "sci-fi"},
        {"id": 2, "vector": [0.21, -0.31, ...],
         "title": "Titanic",   "year": 1997, "genre": "romance"},
    ],
)

# 4. Search with a filter expression
results = client.search(
    collection_name="movies",
    data=[[0.12, -0.44, ...]],          # query vector(s)
    limit=5,
    filter='year >= 2000 and genre == "sci-fi"',
    output_fields=["title", "year", "genre"],
)
for hit in results[0]:
    print(hit["id"], hit["distance"], hit["entity"])
💡 Milvus Lite runs entirely in your Python process — no Docker required. Perfect for learning and prototyping. Switch to the full server later by changing one URI.
